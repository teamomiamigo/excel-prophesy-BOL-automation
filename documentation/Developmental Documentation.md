## Reference

Stable technical notes that don't belong to one changelog entry — add here when something is worth knowing on its own. Keep this short; if it's about the codebase's architecture rather than something learned while fixing a bug, it probably belongs in `CLAUDE.md` instead.

- **Known data-integrity bug (found 2026-07-01, not yet fixed):** 10 pairs of duplicate `bol_records` rows exist in production for the same `technique_trip` (e.g. two rows for `TEC_T_0110814` with the identical `created_at` timestamp down to the microsecond). Root cause suspected in `pull_technique_data()`'s upsert/matching logic — a real trip is getting inserted twice in one pull instead of matched to its existing row. Not caused by anything in this changelog; discovered incidentally while verifying the table-merge below. Tracked as a follow-up, not yet fixed.
- **`start.ps1` encoding gotcha (2026-07-06):** Windows PowerShell 5.1 reads `.ps1` files without a UTF-8 BOM using the system codepage, not UTF-8 — a literal em-dash (`—`) in a comment or string gets misdecoded into a stray quote-like byte, corrupting the parser's quote-tracking for the rest of the file (manifests as unrelated-looking parse errors on far-later lines). Keep `.ps1` files ASCII-only.
- **Inconsistent `python` resolution on this machine (2026-07-06):** bare `python` resolves to different interpreters depending on execution context — an interactive shell (profile loaded) picks one install with packages present; a `-NoProfile` background process (as `start.ps1` spawns) picks a different install (`Python314`) that may be missing packages. Backend deps were installed into both; if `ModuleNotFoundError` recurs after adding a new dependency, install it into both interpreters or pin `start.ps1` to a full path.
- **Postgres native enum gotcha (2026-07-15):** `Base.metadata.create_all()` creates a Postgres `ENUM` type once but never alters it when a Python `enum.Enum` gains a new member later — adding `ActionType.DO_NOT_PAY` required an explicit `ALTER TYPE actiontype ADD VALUE IF NOT EXISTS '...'` in the lifespan migration, run as its own connection/transaction (can't reliably share one with other DDL on older Postgres). Also: the enum label Postgres stores matches SQLAlchemy's enum **member name** (`DO_NOT_PAY`), not `.value` (`"do_not_pay"`) — got this backwards on the first attempt. Any future `ActionType`/`BOLStatus` addition needs the same treatment.

### 2026-07-15 — #64 Pricing calculations — FSC precision, coverage-threshold undercount, and a far more complete rate table
**What:** Three changes to `_apply_access_prog_calc()`/`_parse_alg_csv_context()` (backend/main.py): (1) FSC rate is now derived from the invoice's own exact dollar figures (`fsc_cost_val / alg_freight_total`) instead of the CSV Fuel Surcharge row's own `Rate` label, which is rounded to 2 decimals (e.g. "0.41" when the true rate, confirmed against the matching PDF, is 0.4050). (2) `_RATE_COVERAGE_THRESHOLD` raised from 0.8 to effectively 1.0 (`0.999999`) — previously, a shipment with e.g. 85% of its weight zone-rated would report only that 85% slice's dollars as if it were the whole shipment's cost, no scaling, no fallback, because 85% cleared the old 80% bar; now any unrated zone at all falls back to the invoice's own blended $/cwt for the whole load, same mechanism the code already used for the worse case. (3) New `alg_tariff_rates` table (527 rows, `backend/models.py` `AlgTariffRate`, seeded via `backend/seed_rates.py` from `ALG5_2026_tariff_rates.csv`) sourced from a live query against Prophecy/ShipperPlus's own `dbo.tariff_details`, keyed by the exact `Dest_ID`/`destination_id` format our own pallet data already carries — added as a new priority-2 rate-lookup step in `_apply_access_prog_calc()` (after ALG's own invoiced rate for that exact zone, before the older zip3-keyed `tariff_rates` card), via new `get_alg_tariff_rate()` in `backend/data_layer.py`.
**Why:** Reported symptom was wildly disproportionate Cost % swings from tiny weight differences (1 lb → 5%, 100 lb → 17%). Root-caused against real data this session (two real invoice CSVs recovered from local Outlook cache, cross-checked against a freshly-provided Excel "Verification File" containing the legacy process's own parallel calculation for the same real BOLs): the coverage-threshold undercount was the primary driver of systematic under-pricing across every problem record checked; the FSC rounding was a smaller (~0.5 point) but fully confirmed contributor present on every invoice checked (8/8). Separately, direct measurement found the existing `tariff_rates` card missing 59 of 92 destination zip3s (71.5% of shipped weight) across just 2 real invoices — far worse than the "3 zones" previously documented — closed by importing the far more complete `alg_tariff_rates` table (confirmed 0% missing against the same 2 invoices, and an exact 527/527 correspondence against VisualMail's own `Locations` table when cross-checked via `VM_Locations.xlsx`).
**Files:** backend/main.py (`_parse_alg_csv_context`, `_apply_access_prog_calc`, `_RATE_COVERAGE_THRESHOLD`), backend/data_layer.py (new `get_alg_tariff_rate`), backend/models.py (new `AlgTariffRate`), backend/seed_rates.py (new `load_alg_tariff_rates`, `--alg-tariff` flag), CLAUDE.md (Open Question #11 corrected, access_prog calculation section rewritten, static rate tables list updated)
**Gotcha:** A third, previously-flagged issue (3 specific records whose miss doesn't track weight difference at all, hypothesized to involve minimum-freight-charge/zone-matching interaction) remains genuinely unconfirmed — needs a live per-pallet, per-destination-zone breakdown from Prophecy/ShipperPlus that neither this session nor an earlier one investigating the same issue could pull (no route to that server from this dev environment). Do not extrapolate a fix for it from this entry. Verified the two shipped fixes directly: replayed `_parse_alg_csv_context()` against the real cached CSVs (FSC now derives to 0.404999, not 0.41) and exercised `_apply_access_prog_calc()` with synthetic pallet data confirming the coverage-fallback and new exact-match lookup both fire correctly and produce hand-verified dollar totals — could not verify end-to-end through a live-matched Technique record, since that also requires AWP-SQL-PROD access unavailable here.

### 2026-07-14 — access_prog overstated on minimum-freight-charge zones
**What:** `_parse_alg_csv_context()` now reads ALG's own `Rate` column directly for each freight line instead of deriving it as `Billed$/GrossWt` (falls back to the derivation only if `Rate` is genuinely absent). `_apply_access_prog_calc()`'s ALG-rate pricing branch now also applies the zone's minimum freight charge (`tariff_rates.minimum_freight`, newly surfaced via `get_tariff_rate()`'s return dict) as a floor on the computed cost, matching the floor the internal-card fallback branch already had.
**Why:** Confirmed against 126 real historical invoices (7,290 freight line items) that the `Rate` column is always populated and matches SG360's own internal tariff card exactly, zone for zone — this corrects the 2026-07-10 entry below, whose stated premise ("Rate is blank on every freight row") does not hold against real invoice data (it was true only of a mock test fixture). The real bug: ALG charges a flat minimum fee (commonly $10, sometimes $70/$75) whenever a shipment is small enough that rate×weight would fall below it. Deriving `Billed$/GrossWt` for such a line backs into a fake, inflated "rate" that's only valid for that specific small weight (e.g. a $70 minimum on a 216 lb parcel implies ~$32/cwt against a real $9.26/cwt rate). Reapplying that inflated rate to *our own* (differently sized) pallet weight for the same zone overstated cost — the mechanism behind records like BOL 146824/Z558447 calculating *higher* than ALG's actual bill even when ALG reported *more* weight than SG360's own data. Verified live: our own tariff card's `Minimum Freight` column matches ALG's actual charged minimums exactly for every zone checked (784, 727, 372, 253, 600, 601, 606).
**Files:** backend/main.py (`_parse_alg_csv_context`, `_apply_access_prog_calc`), backend/data_layer.py (`get_tariff_rate` — added `minimum_freight` to its return dict), CLAUDE.md (access_prog calculation section corrected — previously described only the 2026-07-06 state and had not been updated for either the zone-matching fix or the Billed$/GrossWt-derivation fix from 2026-07-10, both below)
**Gotcha:** Not yet backfilled against production — the user is separately wiping and re-uploading all existing records from scratch rather than running `POST /api/admin/recompute-access-prog` against the old data, so no backfill step is needed here. FSC handling was investigated in the same session (CSV shows a rounded 2-decimal fuel surcharge rate, e.g. "0.37", where the invoice's own PDF and dollar figures reflect a more precise "0.365") but was explicitly left unchanged per product owner direction — small (~0.5 point), not the driver of the large misses this fix addresses.

### 2026-07-10 — Fix: access_prog derived from invoice rate (Billed$/GrossWt), not internal tariff table
**What:** Fixed the root cause of all cost_pct values being 200–800% off. `_parse_alg_csv_context()` was reading the `Rate` column to build `alg_rate_by_zip3`, but that column is blank on all ALG freight rows — only the FSC footer has a Rate value. So `alg_rate_by_zip3` was always empty, every record fell through to our internal `tariff_rates` table, which has $1.25–2.23/cwt for local zones (600–606) vs ALG's actual ~$11/cwt — proximity rates that are 5–9× too low. Fix: derive per-zone rate as `Billed$ / (GrossWt/100)` — the actual $/cwt ALG charged, right there in the same row. One 3-line change in `_parse_alg_csv_context()`.
**Why:** Confirmed against live data: Z558234 showed cost_pct ~800% pre-fix because zone 606 looked up at $2.23/cwt while ALG billed ~$11.34/cwt. The fix aligns with the explicit product requirement: rates come FROM the invoice, weights/pallets/pieces come FROM SG360's own data.
**Files:** backend/main.py (`_parse_alg_csv_context` only — 3 lines)
**Gotcha:** The internal `tariff_rates` table is now only used as a last-resort fallback for zones our pallet data covers but the invoice didn't bill — an edge case on well-matched loads. Run `POST /api/admin/recompute-access-prog` after deploy to backfill historical records; records whose original CSV is no longer on disk are reported as `skipped_no_file` and left at their prior (wrong) values.

### 2026-07-10 — Zone matching + 20-day upsert pull + access_prog coverage threshold
**What:** Fixed SCF zone code mismatch: our pallet data uses `destination_zip` (actual postal zip3, e.g. "352") not `destination_id` ("SCF350"), so they could never equal ALG's billed zip3s — 0 of 33 zones rated on a real invoice. Added `_select_canonical_technique_row()` + `_dedupe_technique_rows()` for deterministic dedup (Prepaid+unpooled row wins; load_id resolved via max() across all group rows so a real Type B is never silently downgraded to Type A).
**Why:** Confirmed against real invoice Z558429: all 33 destination zones returned zero rate because the zip3 comparison was always false. Cost % for a Prophecy-matched load with correct weight read 800%+ because 1 of 60 zones rated while the rest dropped, and that single pallet's weight was used as if it were the entire load's. The blended-rate fallback prevents that runaway case.
**Files:** backend/main.py (`_lookup_alg_rate`, `_RATE_COVERAGE_THRESHOLD`, `_apply_access_prog_calc`, `_select_canonical_technique_row`, `_dedupe_technique_rows`, `pull_technique_data`), backend/data_layer.py
**Gotcha:** The ±5 tolerance handles the SCF-code-vs-zip3 naming gap, not genuine missing-zone gaps — a zone with no rate anywhere nearby still stays null. Records refreshed by the 20-day upsert keep their original invoice amounts; only technique-side fields (weight/pallets/pcs/BOL status) are overwritten.
