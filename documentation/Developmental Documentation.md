## Reference

Stable technical notes that don't belong to one changelog entry — add here when something is worth knowing on its own. Keep this short; if it's about the codebase's architecture rather than something learned while fixing a bug, it probably belongs in `CLAUDE.md` instead.

- **Known data-integrity bug (found 2026-07-01) — root cause very likely fixed 2026-07-10, confirmed 2026-07-15, but never marked resolved until now:** 10 pairs of duplicate `bol_records` rows existed for the same `technique_trip` (e.g. two rows for `TEC_T_0110814` with the identical `created_at` timestamp down to the microsecond). The suspected cause — `pull_technique_data()` inserting a real trip twice in one pull instead of matching its existing row — matches exactly what `_dedupe_technique_rows()`/`_select_canonical_technique_row()` (added in commit `ec05ec3`, 2026-07-10, alongside the multi-manifest matching work below) were built to prevent: one real manifest can legitimately produce >1 raw SQL row (the query's own GROUP BY includes fields that can differ pallet-to-pallet), and these collapse them before any upsert. That commit's own changelog entry never mentioned this fix, which is why it sat marked "not yet fixed" for 5 days after being fixed. **Still open**: the 10 already-existing duplicate pairs from before 2026-07-10 were never cleaned up (no backfill script exists for them) — if still present, they'd cause exactly this symptom today for those specific trips, since every pull's exact-match lookup only ever touches one row of each pair. Check directly before assuming this is fully closed out.
- **`start.ps1` encoding gotcha (2026-07-06):** Windows PowerShell 5.1 reads `.ps1` files without a UTF-8 BOM using the system codepage, not UTF-8 — a literal em-dash (`—`) in a comment or string gets misdecoded into a stray quote-like byte, corrupting the parser's quote-tracking for the rest of the file (manifests as unrelated-looking parse errors on far-later lines). Keep `.ps1` files ASCII-only.
- **Inconsistent `python` resolution on this machine (2026-07-06):** bare `python` resolves to different interpreters depending on execution context — an interactive shell (profile loaded) picks one install with packages present; a `-NoProfile` background process (as `start.ps1` spawns) picks a different install (`Python314`) that may be missing packages. Backend deps were installed into both; if `ModuleNotFoundError` recurs after adding a new dependency, install it into both interpreters or pin `start.ps1` to a full path.
- **Postgres native enum gotcha (2026-07-15):** `Base.metadata.create_all()` creates a Postgres `ENUM` type once but never alters it when a Python `enum.Enum` gains a new member later — adding `ActionType.DO_NOT_PAY` required an explicit `ALTER TYPE actiontype ADD VALUE IF NOT EXISTS '...'` in the lifespan migration, run as its own connection/transaction (can't reliably share one with other DDL on older Postgres). Also: the enum label Postgres stores matches SQLAlchemy's enum **member name** (`DO_NOT_PAY`), not `.value` (`"do_not_pay"`) — got this backwards on the first attempt. Any future `ActionType`/`BOLStatus` addition needs the same treatment.

### 2026-07-22 — Phase 6 + 7: revert an accidentally-approved-and-exported record from the Log tab; default sort flipped to oldest-invoice-first

### 2026-07-22 — Invoice auto-matching: removed the shared-budget root cause, then fixed two UX regressions it introduced
**What:** `_process_invoice_csv()` no longer runs a live 90-day wide-fallback Technique search inline during upload/poll — every upload request is now DB-only and fast, and returns `record_id` on both the match and stub branches. The frontend (`uploadInvoiceFiles()`, `handlePollFolder()` in `App.jsx`) automatically fires `POST /api/bols/{id}/retry-match` per new stub right after the upload response comes back (concurrency-limited to 3 at once via a new `runWithConcurrency()` helper), reusing the same search the manual magnifying-glass button already called. `retry_match_invoice()` and `_wide_fallback_technique_search()` are now one shared implementation instead of two independently-written ones (the manual button's old copy had no `query_timeout`/`try-except` around its live queries — fixed as a side effect), and the route's success response now includes `matched_trip`.
**Why:** Root cause of "matches instantly on manual retry but not on upload" was never search reliability — it was the upload-time call sharing its request's timeout budget with everything else already done in that request (CSV parsing, prior files in a batch). Isolating the search into its own request removes the timing problem entirely instead of tuning day-windows/timeouts around it.
**Files:** backend/main.py (`_process_invoice_csv`, `_wide_fallback_technique_search`, `retry_match_invoice`), frontend/src/App.jsx (`uploadInvoiceFiles`, `handlePollFolder`, new `autoRetryNewStubs`/`runWithConcurrency`/`reconcileWithRetryResults`), CLAUDE.md
**Gotcha:** Two real regressions surfaced during live testing of this same change and are fixed in this same commit, not follow-ups: (1) the Invoice Upload/Poll Results summary was built from the immediate per-file response, before the automatic retry-match pass had a chance to run — a record that auto-resolved a moment later still showed "unmatched" in the panel even though the Pending table itself was correct. (2) Fixing that by awaiting the whole automatic pass before showing any results made a batch with several unmatched invoices go completely silent for up to a minute-plus (each live retry-match takes ~13–26s) — no visible progress, which read as broken rather than slow. Final behavior: the results panel shows immediately, with anything still being auto-checked honestly labeled "checking Technique…", then patches in place per-item as the automatic pass resolves each one.

### 2026-07-22 — Blanket 15s query timeout was breaking the main Technique pull
**What:** `_get_connection()` (`backend/data_layer.py`) no longer hardcodes `conn.timeout = 15` for every query on every connection — it now takes an optional `query_timeout` parameter, left unset (pyodbc default: no timeout) unless the caller opts in. `get_technique_data()` and `get_manifest_weights()` gained the same optional parameter, threaded through to `_get_connection()`. `_wide_fallback_technique_search()` (`backend/main.py`) — the one call site the 15s cap was actually built for — now passes `query_timeout=15` explicitly on both of its live calls.
**Why:** First local re-test of `POST /api/admin/pull` since the 2026-07-21 query-timeout fix landed (commit `7f211c0`) failed with `OperationalError: [HYT00] ... Query timeout expired (0)`. That fix set the 15s cap directly inside the shared `_get_connection()` helper, intending to bound only `_wide_fallback_technique_search()`'s live calls within Lambda's 29s budget — but every other live query goes through that same helper, including `get_technique_data()` in `pull_technique_data()` (the main daily pull), whose own live latency was already documented elsewhere in CLAUDE.md as ~15–23s. A flat 15s cap reliably kills that query before it can finish.
**Files:** backend/data_layer.py (`_get_connection`, `get_technique_data`, `get_manifest_weights`), backend/main.py (`_wide_fallback_technique_search`), CLAUDE.md
**Gotcha:** `retry_match_invoice()` and other single-live-call routes that go through `get_technique_data()` (e.g. the 90-day retry-match window) were also silently subject to the old blanket 15s cap and are fixed by the same change, even though they weren't the reported symptom here — worth a quick sanity check on retry-match if it's exercised soon, since it wasn't independently re-verified in this session.

### 2026-07-21 — Cost % formula reversed back to amount / access_prog
**What:** Cost % flipped from `access_prog / amount` back to `amount / access_prog` across all 7 backend write sites in `main.py` (reassign-invoice merge ×2, `_finish_resolving_stub()`, `_apply_invoice_match()` ×2, Wolf/311 stub creation, `recompute-access-prog`), `mock_data.py`'s 10 example records + 1 `flag_reason` string, and the stale comments in `models.py`/`BOLRow.jsx` (both had drifted out of sync during the 2026-07-16 flip — rewritten explicitly here rather than left "accidentally correct").

### 2026-07-21 — Cost-calc accuracy: false ~EST flagging, cost-breakdown tooltip broken on Lambda, wide-fallback query-timeout crash
**What:** Three fixes. (1) `_get_connection()` (`data_layer.py`) now sets `conn.timeout = 15` — pyodbc's query-level timeout, distinct from the connect-phase timeout fixed earlier the same day (see entry below); `_wide_fallback_technique_search()` (`main.py`) wraps its live Technique search in try/except, degrading to a normal `invoice_only` stub instead of an ungraceful Lambda hard-kill. (2) `_apply_access_prog_calc()`'s per-pallet `detail` breakdown is now persisted as JSON on a new `BOLRecord.cost_calc_detail` column at every real calc call site (invoice upload, stub resolution, Wolf/311 stub creation, `recompute-access-prog` backfill); `GET /api/bols/{id}/cost-breakdown` now reads this stored column instead of re-parsing the original invoice CSV from `INVOICE_FOLDER` on every call. (3) `min_charge_uncertain` no longer fires just because `alg_tariff_rates` misses an exact `dest_id` lookup — only when the legacy-table floor actually determined the price, or when no floor info exists anywhere.
**Why:** Re-testing after a full data wipe surfaced three problems at once: one invoice (Z557856) crashed upload with a bare HTTP 500 (CloudWatch showed a 29s Lambda hard-timeout inside the wide-fallback search, no traceback — the connect-phase fix from earlier the same day only bounds the connect, not the query itself); the `~EST` badge showed on nearly every real record, including two (BOL 146500, 146463) independently verified correct against a legacy Excel reference; and the cost-breakdown hover tooltip always said "Cost check unavailable (live mode only, or invoice file not found)" even though mock mode was already off — the real problem was that `INVOICE_FOLDER` is a Windows UNC share the deployed Lambda has no env var for and can never mount regardless, so that route 404'd for every record, always, on the live app. Confirmed fix by recomputing 87 real local records: `~EST` false positives dropped from near-universal to 2 (both legitimate `weight_source_fallback`, 0 `min_charge_uncertain`); BOL 146448 — the record originally flagged as most wrong ($4,484.55 vs. a $5,567.31 reference, ~19% low) — recomputed to $5,668.17 (~1.8% off) with the corrected logic and fresh live data.
**Files:** backend/data_layer.py (`_get_connection`), backend/main.py (`_apply_access_prog_calc`, `_wide_fallback_technique_search`, the 4 real calc call sites, `get_cost_breakdown`, migration block), backend/models.py (`BOLRecord.cost_calc_detail`), frontend/src/components/BOLRow.jsx (`handleCostEnter`, `CostBreakdownPopover`), CLAUDE.md
**Gotcha:** The ambiguous-trip scoring path in `_wide_fallback_technique_search()` still chains two live SQL calls back to back — even with a 15s query timeout each, the combined worst case can still approach Lambda's 29s wall; a real fix needs a per-request deadline budget, not done here. `get_prophecy_data`/`get_alg_invoice` are two more live-SQL call sites in the upload path with no try/except at their call sites either — same bug class, not bundled into this fix. BOL 146448's original ~19% miss was never isolated to one specific root cause before this change — persisting `cost_calc_detail` is what finally made it inspectable, and re-running the calc with the corrected `min_charge_uncertain` logic plus fresh live pallet data happened to resolve nearly all of the gap on this record, but that doesn't rule out the same mechanism recurring on a different real invoice.
